# 权限与安全设计

版本：v1.1  
更新时间：2026-06-29  
适用对象：企业软件工程师 / 架构师 / 技术负责人  

## 1. 本章核心结论

企业 AI 安全设计必须覆盖用户身份、数据权限、工具权限、模型输出、审计追踪和敏感信息保护。

企业 AI 平台的安全边界不能只依赖提示词和模型自律。所有关键权限、确定性判断、敏感数据控制和高风险动作都必须由身份系统、权限系统、规则引擎、配置中心、业务服务和审批流程共同保障。

大模型负责语义理解、生成、归纳和辅助决策；确定性权限判断优先交给规则引擎或业务服务；高风险操作必须经过权限校验、用户确认或审批流程。

## 2. 背景与问题

AI Agent 能访问知识和工具后，风险从“回答错误”升级为“越权访问”和“错误执行”。

### 2.1 背景与建设目标

企业 AI 平台接入知识库、业务系统、MCP 工具和 Workflow 后，安全风险会从传统的页面权限扩展到模型上下文、知识检索、工具调用、自动化流程和生成内容。

常见风险包括：

- 用户通过自然语言绕过页面入口，尝试访问无权数据。
- RAG 检索返回了用户无权查看的知识片段。
- Agent 调用了未授权工具或构造了越权参数。
- 大模型在上下文中泄露敏感字段。
- 高风险写操作被错误执行。
- 审计日志不完整，无法追溯问题来源。

建设目标是形成覆盖“入口、上下文、知识、工具、流程、输出、审计”的全链路安全体系。

### 2.2 安全边界

AI 平台不应绕过企业已有统一认证、组织权限、数据权限和业务审批。Agent 只是新的交互与编排入口，所有业务权限仍应回到企业已有权威系统或规则服务中校验。

## 3. 核心概念

- RBAC：基于角色的权限控制。
- ABAC：基于属性的权限控制。
- Data Masking：敏感字段脱敏。
- Guardrail：输入输出安全约束。
- Least Privilege：最小权限原则。
- Policy Decision Point：权限和策略决策点。
- Policy Enforcement Point：权限和策略执行点。
- Tool Permission：Agent 和用户是否可调用某个工具及其参数范围。
- Prompt Injection：用户或文档内容试图影响模型越权执行的攻击方式。
- Audit Trail：用于追踪安全事件的完整执行记录。

## 4. 应用架构

安全能力应横跨入口、Agent Runtime、知识检索、工具调用和结果输出全过程。

### 4.1 核心架构设计

企业 AI 平台安全架构建议包括：

1. 身份认证层：统一登录、单点登录、Token 校验、会话管理。
2. 权限决策层：RBAC、ABAC、组织权限、数据权限、规则引擎和策略服务。
3. 知识安全层：文档权限继承、知识片段过滤、引用控制和过期知识控制。
4. 工具安全层：工具注册、工具分级、参数校验、调用授权和风险控制。
5. Agent 安全层：上下文过滤、提示词防护、最大步骤限制、人工确认。
6. 流程安全层：Workflow、审批、二次确认和高风险动作拦截。
7. 输出安全层：敏感字段脱敏、引用来源展示、免责声明和转人工。
8. 审计观测层：日志、Trace、指标、告警、回放和安全事件分析。

### 4.2 关键模块说明

- 身份上下文模块：将用户 ID、组织、岗位、角色、数据域和租户信息注入后端执行上下文。
- 权限策略模块：根据角色、属性、资源和动作判断是否允许访问。
- 数据过滤模块：对知识片段、业务查询结果和工具返回结果执行权限过滤。
- 工具安全模块：对工具调用做参数 Schema 校验、权限校验、风险等级判断和超时控制。
- 输出控制模块：对模型回答进行敏感信息检测、脱敏和引用校验。
- 安全审计模块：记录权限判断、规则命中、拒绝原因和高风险操作确认记录。

## 5. 工作流程

认证用户身份，加载权限范围，过滤知识和工具，执行任务，记录审计日志。

### 5.1 业务流程说明

典型安全流程如下：

1. 用户进入 AI 工作台或业务入口，平台完成统一身份认证。
2. 后端解析用户组织、角色、岗位、数据范围和可用 Agent。
3. 用户提交任务后，Agent Runtime 先判断任务类型和风险等级。
4. 权限服务或规则引擎判断用户是否可访问目标知识、工具和业务数据。
5. RAG 检索只返回用户有权查看的知识片段。
6. 工具调用前校验用户权限、Agent 权限、参数范围和操作风险。
7. 高风险动作进入结构化预览、用户确认或 Workflow 审批。
8. 大模型基于授权后的上下文生成回答，输出前进行敏感信息检查。
9. 审计系统记录完整链路，支持安全回放和责任追踪。

### 5.2 权限控制粒度

权限控制至少应覆盖：

- Agent 权限：用户是否可使用某个 Agent。
- 知识权限：用户是否可检索某个知识库、文档或片段。
- 工具权限：用户是否可调用某个工具。
- 参数权限：用户是否可对某个对象、组织、金额或数据范围调用工具。
- 输出权限：用户是否可看到完整结果、脱敏结果或仅提示无权限。
- 流程权限：用户是否可发起、审批、撤回或转交流程。

## 6. 企业案例

薪酬 Agent 必须确保员工只能查看本人薪资，HR 只能查看授权范围内数据。

### 6.1 薪酬 Agent

员工只能查询本人薪资单和公开薪酬政策；HRBP 只能查询授权组织范围；薪酬管理员可查看更完整规则和核算数据。薪资解释必须基于授权后的结构化数据生成。

### 6.2 ERP Agent

采购人员可查询自己负责的采购单和物料信息，但不一定能查看财务付款细节。跨模块查询应由业务服务按数据域过滤后返回。

### 6.3 Office Agent

用户上传文档生成总结时，平台应校验文档权限和分享范围。生成的会议纪要、报告或 PPT 不应自动公开给无权限人员。

## 7. 技术实现建议

所有工具调用都必须在服务端执行权限校验，不能只依赖模型提示词约束。

### 7.1 技术实现建议

- 将权限判断封装为统一服务，Agent、RAG、MCP 和 Workflow 都通过同一套策略校验。
- 工具注册时必须声明资源类型、动作类型、风险等级、参数 Schema 和权限策略。
- 对高风险工具配置二次确认、审批流程或只读限制。
- 对模型上下文进行最小化构造，只注入完成任务所需的授权信息。
- 对提示词注入攻击进行检测，例如忽略系统规则、泄露上下文、绕过权限等指令。
- 对敏感字段建立统一脱敏策略，例如薪资、身份证号、手机号、银行卡、合同金额等。
- 私有化部署场景下，应明确模型、向量库、日志、文件和密钥的部署边界。

### 7.2 权限与安全考虑

- 权限判断必须发生在服务端，前端只负责展示。
- 权限拒绝应返回可解释但不泄露敏感信息的提示。
- Agent 不能因为用户自然语言请求而扩大权限范围。
- RAG 和工具返回的数据应先过滤，再进入模型上下文。
- 审计日志本身也属于敏感数据，应控制访问权限和保留周期。

### 7.3 规则引擎与性能设计

- 规则引擎用于权限策略、数据范围、风险等级、工具路由和审批触发判断。
- 高频权限规则可缓存，但涉及组织变更、离职、敏感授权和高风险操作时应实时校验。
- 权限服务应支持低延迟调用，避免成为 Agent 多步骤执行链路的瓶颈。
- 对模型调用、知识检索、权限判断、工具调用分别设置超时和降级策略。
- 当权限服务不可用时，默认应采用保守策略，拒绝高风险操作或转人工。
- Token 成本控制应与安全相关，避免把大量无关或敏感上下文注入模型。

### 7.4 可观测性与审计

安全审计应记录：用户身份、资源对象、动作、权限策略、规则命中、允许或拒绝结果、工具参数、知识引用、确认记录、审批记录和输出结果。

关键指标包括：越权拦截次数、高风险操作确认率、权限拒绝率、敏感信息脱敏次数、工具调用失败率、异常访问告警数和审计回放完整率。

## 8. 常见问题

问：提示词能否解决越权问题？  
答：不能。提示词只是辅助约束，真正权限必须由系统代码和数据层控制。

问：是否可以把用户权限写进提示词让模型判断？  
答：不建议。权限判断必须由规则引擎、权限服务或业务服务执行，模型只能基于判断结果生成解释。

问：安全策略是否会影响体验？  
答：会增加部分确认和校验步骤，但这是企业生产环境必须承担的成本。可以通过缓存、异步和清晰提示降低体验影响。
