# 提示词与上下文工程

版本：v1.0  
更新时间：2026-06-29  
适用对象：企业软件工程师 / 架构师 / 技术负责人  

## 1. 本章核心结论

提示词工程的核心不是写漂亮指令，而是稳定构造角色、目标、边界、输入、工具和输出格式。

## 2. 背景与问题

企业场景要求输出稳定、可解释、可审计，因此提示词必须与业务规则、权限范围和结构化输出结合。

## 3. 核心概念

- System Prompt：定义角色、边界和全局规则。
- Context Engineering：组织模型推理所需上下文。
- Structured Output：约束模型输出为 JSON、表格或固定模板。

## 4. 应用架构

提示词应由系统模板、业务模板、用户输入、检索结果和工具定义共同组成。

## 5. 工作流程

收集任务信息，筛选知识上下文，注入权限约束，调用模型，校验结构化输出。

## 6. 企业案例

HR Agent 回答政策问题时，应引用制度来源并提示适用范围，不能直接给出越权结论。

## 7. 技术实现建议

把提示词模板版本化管理，并为核心任务建立回归测试样例。

## 8. 常见问题

问：提示词是否可以写死在代码里？  
答：不建议。企业项目应支持模板版本、环境隔离和审计记录。

## 9. 后续延伸
