# 统一画图规范

版本：企业AI平台与Agent应用落地手册 v1.0  
更新时间：2026-06-29  
适用对象：企业软件工程师 / 架构师 / 技术负责人 / 数字化团队  

## 1. 规范目标

本项目后续所有架构设计、Agent 设计、RAG 设计、Workflow 设计、业务 Agent 方案、汇报材料，都应采用统一画图规范。

统一画图规范的目标是让不同读者可以从不同角度理解系统：

- 管理者看总体结构和能力边界。
- 架构师看系统模块和部署方式。
- 开发人员看服务交互和接口调用。
- 业务人员看业务流程和 AI 介入位置。
- 运维人员看部署、监控、链路追踪和异常处理。

## 企业AI标准体系图

企业 AI 标准体系图属于最高层级图，用于描述企业 AI 整体设计约束体系，优先级高于所有架构图、流程图、时序图、数据流图、领域模型图和部署图。

企业 AI 标准体系图必须出现在项目汇报第一页，用于明确本项目的五大标准支柱：架构标准、行为标准、数据标准、治理标准和工程标准。它回答的是“所有企业 AI 设计必须遵守什么标准”，而不是“某个系统如何运行”。

总控架构图、模块视角图、Agent 图、MCP 图、RAG 图、Workflow 图和业务链路图都必须接受企业 AI 标准体系图约束。

## 总控架构图（最高级图）

每个企业 AI 平台只能有一张总控架构图。总控架构图用于战略汇报、架构评审和方案总览，是本项目架构图体系的最高级图。

总控架构图回答的是“企业 AI 平台如何在统一治理下运行”，必须体现用户交互层、AI 能力层、治理控制层、模型与算力层、企业业务系统层，以及权限前置、规则优先、全链路审计等关键原则。

其他所有图，包括 Agent、MCP、RAG、Workflow、Multi-Agent、数据流图、领域对象模型图、部署图和业务链路图，都必须向总控架构图对齐。不允许多个“全局架构图”并存；旧“总体架构图”统一降级为模块视角图，用于说明平台模块组成，不再作为全局唯一图。

## 3. 重要章节配图原则

以后每一个重要章节至少应配套多张图，形成固定模板，而不是只画一张总架构图。

例如《第 3 章：Agent》建议统一包含：

1. 静态架构图：说明 Agent 的组成，例如 Planner、Memory、Tool、Reflection、Agent Runtime 等。
2. 动态 Workflow 图：说明 Agent 从接收任务到完成执行的完整流程。
3. 时序图：说明 Agent 与 LLM、MCP Server、企业系统之间的交互过程。
4. 部署架构图：说明 Agent 在企业环境中的部署方式，例如 Gateway、Agent Service、向量库、大模型、GPU、监控平台等。

重要章节的配图应覆盖“是什么、怎么运行、如何交互、如何部署、如何审计”几个层面。

## 5. 图命名规范

Markdown 内标题建议：

- `## 架构图`
- `## 流程图`
- `## 时序图`
- `## 数据流图`
- `## 部署图`
- `## 领域对象模型图`

Mermaid 源文件命名建议：

```text
章节名称-图类型.mmd
```

示例：

- `企业AI平台总体架构图.mmd`
- `Agent-Runtime执行链路图.mmd`
- `用户订单结算财务协同链路图.mmd`
- `订单Agent领域对象模型图.mmd`
- `订单异常识别与处理建议时序图.mmd`

## 6. Mermaid 编写要求

1. 优先使用 Mermaid。
2. 图中节点命名使用中文，必要时保留英文技术名。
3. 每张图控制在 8 到 15 个核心节点。
4. 不要为了完整而画得过度复杂。
5. 图中需要体现关键边界，例如用户入口、Agent Runtime、规则引擎、权限中心、RAG、MCP、Workflow、业务系统、审计日志、可观测性。
6. 高风险业务流程图中必须体现权限校验、人工确认、审批和审计。
7. Mermaid 源文件需要同步保存到 `mermaid` 目录。
8. Markdown 文档中也需要嵌入 Mermaid 代码块。
9. 数据流图应重点说明数据来源、数据处理、权限过滤、系统调用、规则判断、审计留痕和异常降级，不应只画系统模块连接关系。

## 8. 执行要求

后续新增或深化重要章节时，应先判断读者对象，再选择图类型。若章节涉及高风险业务动作，图中必须体现权限校验、规则引擎、人工确认、审批、审计日志和可观测性。
